## 1. Pioneer tag carve-out for platform owners

**Why it exists.** Pioneer = "first wallet to earn this achievement" — a DB-only tag (no on-chain bit) that's meant to reward genuine first-mover community users. Jon's testing activity keeps claiming the Pioneer tag on achievements he triggers during dev/QA, which locks the community out of that status for those achievements permanently.

**State right now.**

- `wordy` (token #22) Pioneer retroactively cleared 2026-04-20 (minted as Pioneer by mistake earlier that session; clear-then-let-next-winner-claim pattern used).

- Memory file `feedback_notabot-no-pioneer.md` is the interim behavioural guard for Claude Code across sessions. Code-level enforcement is still needed.

**Implementation options.**

| Option | How | Pros | Cons |
|---|---|---|---|
| **A. New env var** `PIONEER_EXCLUDED_WALLETS` | Added to `ecosystem.config.cjs`; `awardAchievement()` reads list and skips the pioneer-tag block when wallet matches | Consistent with `OWNER_WALLETS` / `BUILDER_WALLETS` patterns; list editable without code change; explicit and auditable | One more env var to remember; needs PM2 restart to change |
| **B. Reuse `OWNER_WALLETS`** | `awardAchievement()` checks against existing `OWNER_WALLETS`; if match, no pioneer | No new config surface | Semantically mixes two concerns — `OWNER_WALLETS` currently means "bypass mint fees", now would also mean "can't be pioneer". If future owner needs pioneer-eligibility that would diverge. |
| **C. Hardcoded constant** | `const PIONEER_EXCLUDED = [...]` in `db/index.mjs` | Simplest | Requires code change to update; no separation of config from code |

**Open questions for PC:**
1. Does the rule apply to *all* owner wallets, or only `notabot.qf`? (The Ledger and `0x08ba…70C3` also bypass mint fees today. Do we want them excluded from Pioneer too?)
2. `bug-hunter` is carved out by history. Does the code-level guard need a per-achievement allow-list (e.g. "these specific pairs may be Pioneer"), or is "don't retroactively fix, just stop future ones" good enough and future carve-outs get handled by manual DB writes like today?
3. Is there a case for awarding Pioneer to owner wallets when *they are the first real user-style interaction*, vs. being excluded purely on identity? (E.g. a game that launches where Jon's playtest is the only early activity.)

**Claude Code's tentative recommendation:** Option A (new env var). Start with just `notabot.qf` in the list; expand if/when other owners demonstrate the same pattern. Does not touch the `bug-hunter` row. Carve-outs for future edge cases handled by DB writes, not code.

## 2. Maffsy free-play achievement/leaderboard gate

**The problem.** `qf-dapp/games/maffsy/index.html:933`:

```js
if (PREFS.free) return;  // short-circuits BEFORE POST to /maffsy/complete
```

Free play never hits the backend. Consequences:
- `wordy` (≤3 guesses) and `binary-decision` (=2 guesses) never award
- `feel-no-pressure` (50-game clean streak) never accrues
- `the-novelist` (6-guess loss) never awards
- Streak counter never increments on the server
- Leaderboard prompt (current-streak metric) never fires

This bug was **pre-existing**. The 2026-04-20 default-flip to free play for new users amplified exposure — a new user today never gets any Maffsy achievement unless they manually toggle off free play.

**The tension.** `recordMaffsyResult()` at `db/index.mjs:1763` uses:

```sql
INSERT INTO maffsy_streaks (wallet, play_date, won, guesses, word_id)
VALUES (?, ?, ?, ?, ?)
ON CONFLICT(wallet, play_date) DO UPDATE SET won = excluded.won, ...
```

Daily mode = one row per day, the last write is the definitive daily result. Free play = unlimited plays per day, and blind-firing the endpoint every game would clobber today's row repeatedly and corrupt streak semantics (imagine: win at 09:00 sets today=win, streak increments; lose at 10:00 sets today=loss; streak breaks even though you already won today).

**Implementation options.**

| Option | Shape | Pros | Cons |
|---|---|---|---|
| **A. Split endpoints** | `POST /maffsy/achievement-check` (fires always) + `POST /maffsy/daily-complete` (daily mode only) | Clean separation; no conditional logic server-side | Two round trips from client; more surface to keep in sync |
| **B. Single endpoint with `mode` flag** | `POST /maffsy/complete` takes `{ won, guesses, mode: 'daily' \| 'free' }`; server always runs achievement checks, but only writes `maffsy_streaks` when `mode='daily'` | Single endpoint, minimal client change | Still have the awkward semantic that "free play doesn't contribute to streak" — needs UX decision |
| **C. Smart `recordMaffsyResult`** | Server writes `maffsy_streaks` for free play too, but *only if no row exists yet for today OR today's existing row is not a win*. So first play of the day lands; subsequent wins are ignored (already a win); losses only stick if today had no win yet. | No client change; daily and free play both contribute naturally | Subtle logic, harder to reason about; "one win per day counts" is an invisible rule |
| **D. Separate table** `maffsy_free_plays` | Free play writes to a new table tracking all attempts; achievement checks read from both | Cleanest data model | Most code; duplicates streak/stats logic |

**Open questions for PC:**
1. **Leaderboard semantics.** Jon has said "the leaderboard was supposed to be for a max streak anyway." In a mixed daily + free world, what's the streak? Options: (i) daily-only (one-per-day, clean definition, excludes free); (ii) mixed (free-play wins count, but only one per day — option C); (iii) unified free-play streak (a separate streak per mode). Choosing this drives the whole design.
2. **`feel-no-pressure` (50 wins).** Counter is `wallet_stats.maffsy_clean_streak`. Does this increment on free-play wins, or daily only? Consistency with the leaderboard answer.
3. **`the-novelist` (6-guess loss).** Currently requires a recorded loss. If free-play losses don't hit the server, this is only achievable in daily mode. Is that fine, or should it be cross-mode?
4. **UX for revealing the free-mode-has-no-achievements reality.** If we decide free play is silent by design, should the modal have a "switch to daily to earn achievements" prompt? Or is the whole point of free play to be zero-stakes and zero-reward?

**Claude Code's tentative recommendation:** Option C + mode=free added client-side. Rationale: keeps one endpoint; "one-win-per-day" naturally matches the spirit of a daily streak; free-play wins contribute to discoverable progress (feel-no-pressure, wordy, etc.) without changing daily semantics. But this depends entirely on PC's answer to Q1 above.

## 3. Box 1 drift audit

**State.** Box 1 `/home/jon/mathswins-dapp/` is not a git repo. Deploys happen via `scp` per-file. Over time this has accumulated orphan files that don't exist in the GitHub clone at `/home/jon/MathsWins/dapp-backend/`:

**Files at `src/` root** (5 extras):
- `api.mjs` — superseded by `routes/api.mjs`
- `index.mjs` — superseded by `server.mjs`
- `estimation-engine.mjs` — superseded by `games/estimation-engine.mjs`
- `kenken.mjs` — superseded by `games/kenken.mjs`
- `games/scoring.mjs` — alongside the real `scoring.mjs` at src root (which is still current)

**Nested duplicate tree at `src/games/games/`** (16 extras): all 16 game files re-copied one directory deeper than they should be. Almost certainly a deploy misstep where `scp -r games/ /home/jon/mathswins-dapp/src/games/` was run when `/home/jon/mathswins-dapp/src/games/` already existed.

**Risk if we just delete.** None of these appear in the imports of the current source (`server.mjs`, `routes/api.mjs`, `league-settle.mjs`, `escrow.mjs`, `scoring.mjs`, `telegram.mjs`, `achievement-checker.mjs` all use the canonical paths). But "appear" is not "proven" — confirmation requires a recursive grep of every file against every orphan path. Doable in one pass but needs care.

**Risk if we leave them.** Future deploys could shadow the live code accidentally. Also, any future Claude/developer looking at the tree has to mentally filter noise. And the fact that `scp` creates nested duplicates silently means this will keep happening unless we establish a safer deploy pattern.

**Implementation options.**

| Option | Shape | Pros | Cons |
|---|---|---|---|
| **A. Audit-then-delete** | `grep -r` each orphan path across Box 1 codebase, confirm zero imports, delete | Safe, conservative | Time: 30-60 min; needs careful execution |
| **B. Rsync from local** | `rsync -av --delete /home/jon/MathsWins/dapp-backend/src/ jon@box1:/home/jon/mathswins-dapp/src/` | One command, known end-state | `--delete` is aggressive; any file only-on-Box-1 that IS needed would be wiped with no recovery path (Box 1 isn't a git repo) |
| **C. Live with it + document** | Add to `docs/known-issues.md`, move on | Zero risk right now | Tech debt compounds; future deploys may duplicate again |
| **D. Make Box 1 a git repo** | `git init` on `/home/jon/mathswins-dapp/`, commit current state, deploy via `git pull` going forward | Clean future; rollback possible; drift visible via `git status` | Big one-time migration; must decide how to handle `.env`, `ecosystem.config.cjs`, `data/`, `logs/` (all gitignored); deploy flow changes (scp → push) |

**Open questions for PC:**
1. What is the **intended** deploy model? Status quo (scp individual files) has produced this drift over months. Would it be safer long-term to make Box 1 a git repo (option D) and deploy via `git pull` from the authoritative GitHub clone? That's the pattern scribble uses and it hasn't drifted.
2. If we go with option A or D, is there an intermediate sanity check — e.g. diff-everything-against-GitHub, commit the diff to a temporary branch, review, merge? Or trust the audit and move?
3. `data/` and `logs/` and `.env` and `ecosystem.config.cjs` — these stay Box-1-only regardless. Are we comfortable leaving `ecosystem.config.cjs` ungit-tracked given we routinely edit it (OWNER_WALLETS, FOUNDING_MEMBER, etc.) and those edits currently vanish if Box 1 dies?

**Claude Code's tentative recommendation:** Option A now (delete the obvious orphans safely after audit), Option D later (migrate deploy to git-pull pattern for long-term hygiene). Option D deserves its own design session — it touches rollback strategy, secret handling, and deploy discipline.
